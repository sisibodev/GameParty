> 버전: v0.1 (초안)
> 작성일: 2026-04-12

---

## 1. 프로젝트 개요

### 목적
웹 기반 멀티 게임 플랫폼을 구축하여, 허용된 사용자만 접근 가능한 개인 게임 포털을 운영한다.

### 핵심 컨셉
- 메인 페이지에서 게임 목록을 카드/아이콘 형태로 제공
- 카드 선택 시 해당 게임으로 이동하여 즉시 플레이
- 기본적으로 비공개이며, 허용된 사용자만 접근 가능
- 게임별 개별 설계 및 추가 가능한 확장성 있는 구조

---

## 2. 서비스 구조

```
게임 플랫폼
├── 접근 제어 (인증/인가)
│   ├── 로그인 페이지
│   └── 허용된 사용자 관리
├── 메인 페이지 (게임 로비)
│   ├── 게임 카드/아이콘 목록
│   ├── 게임 검색 및 필터 (추후 확장)
│   └── 사용자 정보 표시
└── games/ (게임별 개별 설계)
    ├── 주식 보드게임/
    ├── 게임 B/
    └── 게임 N/...
```

---

## 3. 페이지 설계

### 3.1 로그인 페이지

| 항목 | 내용 |
|------|------|
| 경로 | `/login` |
| 접근 | 누구나 (비인증 상태에서 기본 진입점) |
| 기능 | 사용자 인증, 비인가 접근 시 리다이렉트 |

**구성 요소**
- 로그인 폼 (ID / 비밀번호 또는 초대 코드 방식 - 추후 결정)
- 플랫폼 로고 및 소개 문구
- 비인가 사용자에게는 추가 정보 노출 없음

---

### 3.2 메인 페이지 (게임 로비)

| 항목 | 내용 |
|------|------|
| 경로 | `/` 또는 `/lobby` |
| 접근 | 허용된 사용자만 |
| 기능 | 게임 목록 표시, 게임 선택 및 진입 |

**구성 요소**
- 상단 헤더: 플랫폼 이름, 사용자 정보, 로그아웃 버튼
- 게임 목록 영역
  - 카드 형태: 게임 썸네일, 게임 이름, 간단한 설명
  - 아이콘 형태: 콤팩트한 그리드 레이아웃 (선택 가능하도록 토글 - 추후 고려)
- 카드 클릭 시 해당 게임 페이지로 이동

**게임 카드 정보 구성 (예시)**
```
┌─────────────────┐
│  [썸네일 이미지]  │
│                 │
│  게임 이름       │
│  짧은 설명       │
│  [플레이 버튼]   │
└─────────────────┘
```

---

### 3.3 개별 게임 페이지

| 항목 | 내용 |
|------|------|
| 경로 | `/game/{game-id}` |
| 접근 | 허용된 사용자만 |
| 기능 | 개별 게임 플레이 |

- 각 게임은 별도로 설계 및 기획 진행
- 공통 레이아웃(헤더, 뒤로가기 등)은 공유 컴포넌트로 처리
- 게임 목록은 설정 파일 혹은 DB로 관리하여 쉽게 추가 가능

## 4. 접근 제어 설계

### 기본 정책
- 기본적으로 **비공개** 플랫폼
- 허용된 사용자만 로비 및 게임에 접근 가능
- 비인가 접근 시 로그인 페이지로 리다이렉트

### 인증 방식 ✅ 확정 — Google OAuth + 허용 이메일 목록

- **Firebase Auth**를 통해 구글 로그인 처리
- 로그인 후 해당 구글 계정 이메일이 **허용 목록에 있는지 확인**
- 목록에 없으면 접근 거부 및 안내 메시지 표시
- 사용자 추가/제거는 허용 이메일 목록만 수정하면 됨

**사용자 입장 경험**
```
접속 → 구글 로그인 버튼 클릭 → 구글 계정 선택 → 로비 입장
```

**관리자(운영) 입장**
```
지인 추가: 이메일 주소 받아서 허용 목록에 추가
지인 제거: 허용 목록에서 이메일 삭제
```

### 세션 관리
- Firebase Auth 기본 세션 유지 사용
- 로그아웃 기능 제공 (헤더에 버튼 배치)

## 5. 기술 스택 ✅ 확정

| 분류 | 선택 | 비고 |
|------|------|------|
| 프론트엔드 프레임워크 | **React** | 플랫폼 전반 UI, 라우팅, 인증 처리 |
| 2D 게임 렌더링 | **PixiJS** | 대부분의 2D 게임에 적용 |
| 3D 게임 렌더링 | **Three.js** | 3D가 필요한 게임에만 선택적 적용 |
| 인증 | **Firebase Auth** | Google OAuth + 허용 이메일 목록 방식 |
| 데이터베이스 / 기록 | **Firebase Firestore** | 게임별로 기록 저장 여부 선택 적용 |
| 실시간 멀티플레이 (기본) | **Firebase Realtime Database** | 턴제 등 간단한 실시간 동기화 |
| 실시간 멀티플레이 (확장) | **Node.js + Socket.io** | 빠른 실시간 처리가 필요한 게임에 적용 |
| 호스팅 (프론트) | **Firebase Hosting** | React 앱 배포 |
| 호스팅 (게임 서버) | 미정 | Node.js 서버 필요 시 결정 |

### 렌더링 전략
- 게임 렌더링은 React 내부에서 `<canvas>` 기반으로 처리
- 2D 게임 → PixiJS
- 3D 게임 → Three.js
- 플랫폼 구조(React) 변경 없이 게임별로 라이브러리 선택 적용 가능

### 서버 전략
- **기본**: Firebase만으로 운영 (별도 서버 없음)
- **확장**: 빠른 실시간 처리나 서버 로직 검증이 필요한 게임이 생기면 Node.js + Socket.io 서버 추가
- 게임별로 Firebase 방식 / Socket.io 방식 선택 적용 가능

### 데이터베이스 전략
- 게임별로 기록 저장 여부를 독립적으로 결정
- 기록 저장 O → Firestore에 게임별 컬렉션으로 관리
- 기록 저장 X → DB 연동 없이 게임만 구현

## 6. 게임 목록 관리

### 게임 메타데이터 구조 (예시)

```json
{
  "id": "game-001",
  "name": "게임 이름",
  "description": "게임에 대한 짧은 설명",
  "thumbnail": "/images/game-001.png",
  "path": "/game/game-001",
  "enabled": true,
  "tags": ["퍼즐", "1인용"]
}
```

### 게임 추가 프로세스
1. 게임 개별 기획 및 개발
2. 게임 메타데이터 등록 (목록에 카드 추가)
3. 게임 경로 라우팅 연결
4. 테스트 후 `enabled: true` 로 활성화

## 10. 새 게임 기획 시작 방법

새 게임을 기획할 때는 **새 Claude 프로젝트**를 열어서 진행한다.
메인 기획서(이 파일)를 첨부해서 플랫폼 전체 맥락을 Claude에게 전달한 뒤 게임 기획 MD를 작성한다.

### 시작 절차

**1단계 — 새 Claude 프로젝트 생성**
- claude.ai에서 새 프로젝트 생성
- 프로젝트 이름 예시: `게임플랫폼 - [게임 이름]`

**2단계 — 메인 기획서 첨부 후 아래 문구로 시작**
```
이 파일은 게임 플랫폼 메인 기획서야.
이 플랫폼에 들어갈 [게임 이름] 게임 기획 MD를 새로 만들어줘.
```

**3단계 — 게임 기획 MD 작성**
- 아래 템플릿 항목 기준으로 작성
- 완성된 MD 파일은 Docs 폴더에 저장

**4단계 — 메인 기획서 업데이트**
- 위 게임별 기획 현황 표에 게임명 + 기획서 링크 추가
